# AGENTS.md

## Загальна інформація про проєкт

Цей проєкт є основою дипломної роботи на тему:

«Розподілена інформаційна система управління життєвим циклом та безпекою зберігання медичних препаратів»

Поточний проєкт виріс із курсової роботи. У ньому вже присутні backend, web-частина, mobile-частина, логіка авторизації, ролі користувачів, журнал аудиту, IoT/датчики, моніторинг умов зберігання, сповіщення та фонові сервіси.

## Головна мета роботи

Мета полягає не в повному переписуванні системи з нуля, а в:

- проведенні аудиту поточного стану проєкту;
- збереженні корисної бази з курсового;
- розвитку системи до рівня дипломної роботи;
- поступовому розширенні функціоналу відповідно до теми диплома.

## Мова роботи

Усі пояснення, аналітичні документи, плани, висновки та службові markdown-файли потрібно вести українською мовою.

Назви класів, методів, файлів, папок, таблиць, DTO, API endpoint'ів та інших технічних сутностей потрібно залишати в оригінальному вигляді, як вони названі в коді.

Не змішувати без потреби українську й англійську в одному реченні, окрім випадків, коли згадується конкретна технічна назва з проєкту.

## Обов’язковий порядок роботи

Перед початком будь-якої великої задачі потрібно обов’язково прочитати:

1. `PROJECT_CONTEXT.md`
2. релевантні файли проєкту
3. поточні конфігурації, моделі, сервіси, контролери та інші пов’язані частини системи

Не можна вигадувати функціонал, якого немає в коді.
Потрібно чітко розрізняти:

- що вже реалізовано;
- що реалізовано частково;
- чого немає;
- що потребує рефакторингу;
- що потребує повної переробки.

## Правила аудиту та аналізу

Під час аналізу потрібно охоплювати всю систему:

- backend;
- database / entities / models / migrations;
- controllers / API;
- services / business logic;
- authentication / authorization / roles;
- audit / logging;
- background services;
- IoT / датчики / телеметрію;
- web frontend;
- mobile app;
- конфігурації;
- тести;
- документацію.

Будь-які висновки мають бути прив’язані до реального коду і файлової структури.
Якщо якась функція реалізована лише частково, це потрібно прямо зазначати.

## Дисципліна ведення документації

Після завершення будь-якої суттєвої задачі потрібно оновлювати `PROJECT_CONTEXT.md`.

У `PROJECT_CONTEXT.md` необхідно фіксувати:

- що було проаналізовано;
- що було змінено;
- які рішення прийнято;
- які технічні борги виявлено;
- які ризики існують;
- що потрібно робити далі.

## Пріоритети розробки

Пріоритет реалізації:

1. backend і предметна модель
2. база даних і зміни схеми
3. API
4. web frontend
5. mobile app
6. тестування
7. розгортання, документація, діаграми

## Напрямок цільової архітектури диплома

Майбутня дипломна система повинна підкреслювати:

- backend API;
- окремий сучасний web frontend;
- mobile app;
- database;
- IoT / sensor subsystem;
- notifications / background services;
- audit / security;
- логіку життєвого циклу препарату;
- логіку безпеки зберігання;
- інциденти зберігання;
- розподілену архітектуру системи.

## Важливі обмеження

- Не переписувати все без потреби.
- Віддавати перевагу поетапному рефакторингу, а не повному знесенню існуючого рішення.
- Усі нові рішення повинні підтримувати бізнес-логіку предметної області.
- Опціональні функції типу AI, PostgreSQL, Redis, QR/OCR, push або реальний фізичний IoT не повинні блокувати реалізацію основного функціоналу диплома.

## Обов’язкові файли проєктної пам’яті

Потрібно підтримувати в актуальному стані такі файли:

- `PROJECT_CONTEXT.md` — живий контекст проєкту, рішення, ризики, висновки
- `AUDIT_AND_DIPLOMA_PLAN.md` — технічний аудит і план розвитку
- `IMPLEMENTATION_ROADMAP.md` — покроковий план реалізації
- за потреби інші службові markdown-файли, якщо вони прямо підтримують дипломну розробку

## Формат прийняття рішень

При виборі між кількома варіантами потрібно:

1. оцінити, що вже є в коді;
2. обрати варіант із найменшим ризиком зриву строків;
3. віддавати перевагу рішенням, які реально встигнути реалізувати в межах дипломної роботи;
4. окремо позначати optional-рішення, які можна відкласти.

## Що вважати успішним результатом

Успішний результат будь-якого етапу — це не просто аналіз, а:

- зрозумілий документований висновок;
- зафіксоване рішення;
- конкретний наступний крок;
- мінімум невизначеності для подальшої розробки.
