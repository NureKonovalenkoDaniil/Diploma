# IoT Device Provisioning & Security

## Проблема

При реєстрації IoT-пристрою в системі необхідно забезпечити, щоб:

1. Девайс отримав унікальний `deviceSecret` для подальшої аутентифікації.
2. Неавторизовані особи не могли "перехопити" або "підмінити" пристрій.
3. Процес був практичним для реальних сценаріїв розгортання.

Поточна реалізація (claim-flow) дозволяє будь-якому, хто знає `DeviceId`, викликати `/api/iotdevice/claim` і отримати `deviceSecret` — це критична вразливість.

---

## Розглянуті варіанти рішення

### Варіант 1: Factory Provisioning (рекомендовано)

**Основна ідея:**  
На етапі **виробництва** (або перед розгортанням) кожен пристрій отримує:

- Унікальний `DeviceId` (серійник або надрукований код).
- **Bootstrap token** (одноразовий тимчасовий ключ), який "вшито" у прошивку або збережено у NVS (non-volatile storage).

**Процес використання в системі:**

1. Адмін реєструє пристрій у UI, вводячи лише `DeviceId`.
2. UI надсилає `POST /api/iotdevice/register` з `DeviceId`.
3. Сервер створює запис пристрою у БД (без `deviceSecret`), генерує **очікуваний bootstrap token** (або просто записує, що цей девайс "можна провізіонити").
4. Девайс на борту вже має цей bootstrap token → робить `POST /api/iotdevice/claim` з `DeviceId + BootstrapToken`.
5. Сервер перевіряє збіг → видає `deviceSecret`.
6. Девайс зберігає `deviceSecret` у локальній NVS (на подальші запити не потребує bootstrap token).

**Переваги:**

- Жоден ручний ввід коду на місцеці розгортання.
- Неможливо "перехопити" пристрій, якщо не маєш bootstrap token.
- Масштабується для сотень/тисяч пристроїв.
- Реалістично для виробничого сценарію.

**Недоліки:**

- Потрібна організація етапу "підготовки" девайсів перед розгортанням.
- Bootstrap token має бути збережений безпечно (у NVS, не у open config).

**Для дипломної демонстрації:**  
Можна пояснити архітектурно, але технічно реалізувати спрощено: покласти bootstrap token у [IoTEmulate/src/config.h](IoTEmulate/src/config.h) і розповісти, як це працювало б на справжньому пристрої.

### Варіант 2: Claim Token (одноразовий код в UI)

**Основна ідея:**  
Кожен раз, коли адмін реєструє нове пристрій, сервер генерує **短暂的одноразовий код** (наприклад, 6–8 символів, дійсний 5–10 хвилин).

**Процес використання в системі:**

1. Адмін у UI реєструє пристрій (`POST /api/iotdevice`).
2. Сервер повертає:
   - `DeviceId`
   - **`ClaimToken`** (наприклад `A7K9M2`)
   - Мета-інформація про пристрій.
3. Адмін вводить `DeviceId` і `ClaimToken` у конфіг девайса (або передає їх до девайса якимось каналом: USB, серійна консоль, BLE, мобільний додаток).
4. Девайс робить `POST /api/iotdevice/claim` з `DeviceId + ClaimToken`.
5. Сервер перевіряє:
   - `ClaimToken` існує в БД і не закінчився.
   - `DeviceId` збігається.
   - Видає `deviceSecret`.
6. ClaimToken позначається як "використаний" (або видаляється).

**Переваги:**

- Простіше реалізувати: не потрібна попередня підготовка девайсів.
- Мінімальні вимоги до девайса (просто знати `DeviceId` и `ClaimToken`).
- Для демонстрацій та лабораторних сценаріїв тісно інтегрується з UI.

**Недоліки:**

- Потрібна взаємодія адміна і девайса (вввід коду десь локально).
- Якщо код витече — хто-небудь може перехопити пристрій в це вікно часу.
- Неправильно масштабується для тисяч пристроїв, що розгортаються паралельно.

**Для дипломної демонстрації:**  
Можна тестувати локально (UI → березь код → вводиш у `config.h` емулятора → перезапускаєш). На захисті демонструється як "локальна реєстрація", можна пояснити, як це розширити на BLE/USB.

### Варіант 3: Claim Window (часове вікно без коду)

**Основна ідея:**  
Після створення пристрою в UI сервер дозволяє claim **тільки протягом N хвилин** (наприклад 2 хвилини). Девайс має встигнути за цей час.

**Процес використання в системі:**

1. Адмін реєструє пристрій у UI.
2. UI показує: "Пристрій готовий до підключення. У вас є 2 хвилини, щоб запустити девайс".
3. Девайс робить `POST /api/iotdevice/claim` з просто `DeviceId` (без будь-яких кодів).
4. Сервер перевіряє:
   - `DeviceId` існує в БД.
   - Пристрій ще в "claim window" (створений менше 2 хвилин тому).
   - Видає `deviceSecret`.
5. Після закриття вікна claim блокується.

**Переваги:**

- Без введення коду (просто встигнути запустити девайс).
- Просто для користувача (клац у UI → запустив девайс → готово).

**Недоліки:**

- Менше безпеки: будь-хто, хто бачить ліст пристроїв у вікні часу, може зробити claim.
- Вузьке вікно часу ускладнює реальні сценарії (девайс розгортається давно після реєстрації).

**Для дипломної демонстрації:**  
Найпростіший варіант, але менш реалістичний для "справжнього світу".

## Рекомендація для дипломної роботи

**Технічно реалізувати:** **Варіант 1 (Factory Provisioning)** — це найреалістичніше і найбезпечніше.

**Архітектурно пояснити:**

- На виробництві девайси отримують унікальні bootstrap token.
- У системі адмін реєструє девайс за `DeviceId`.
- Девайс автоматично робить claim без ручних дій.

**У коді для демонстрації:**

- Покласти bootstrap token у [IoTEmulate/src/config.h](IoTEmulate/src/config.h) як `BOOTSTRAP_TOKEN`.
- Реалізувати на сервері перевірку `BootstrapToken` при claim.
- На захисті показати, як це масштабується.

**Альтернатива (якщо Factory Provisioning складно):**  
Варіант 2 (Claim Token) як "спрощена версія для лаб", з поясненням, як це розширити у "справжній світ".
