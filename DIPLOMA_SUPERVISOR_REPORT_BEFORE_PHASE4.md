## 1) Ідея дипломного проєкту

Проєкт розвиває курсову роботу до рівня **розподіленої інформаційної системи** для:

- управління життєвим циклом медичних препаратів;
- контролю безпеки умов зберігання;
- автоматичного виявлення та ведення інцидентів зберігання;
- централізованого аудиту подій;
- роботи через API + web-клієнт + IoT-підсистему.

Ключовий акцент диплома: не переписувати систему з нуля, а еволюційно розширити наявне рішення до предметного рівня дипломної роботи.

## 2) Що вже зроблено (підтверджено кодом)

## 2.1 Backend / API / безпека

- Реалізовано ASP.NET Core 8 API з JWT Bearer та ролями (`Administrator`, `User`, `Sensor`).
- Виправлено критичні проблеми Фази 1:
  - `GenerateJwtToken`: async/await без `.Result`;
  - термін JWT читається з `Jwt:ExpireDays`;
  - ключ JWT прибрано з `appsettings.json` (порожнє `Jwt:Key` у репозиторії, використовується User Secrets);
  - `StorageConditionController` захищено `[Authorize(AuthenticationSchemes = JwtBearerDefaults.AuthenticationScheme)]`;
  - додано `GET /api/auth/me`.

## 2.2 Предметна модель і база даних

Після Фази 2 у схемі є:

- нові сутності: `StorageLocation`, `StorageIncident`, `MedicineLifecycleEvent`, `Notification`;
- розширення `Medicine` (виробник, партія, опис, температурні межі, FK на локацію);
- розширення `AuditLog` (`EntityType`, `EntityId`, `Severity`);
- оновлений `MedicineStorageContext` і міграція `20260413203011_DiplomaPhase2_DomainModel`.

## 2.3 Бізнес-сервіси і автоматизація

- Є 8 сервісів із інтерфейсами (`IService*`) у шарі `Services`.
- Фаза 3 завершена:
  - `StorageConditionMonitoringService` створює/закриває `StorageIncident`, пише `Notification` і `AuditLog`, працює з інтервалом із конфігурації;
  - `ExpiryNotificationService` формує сповіщення про закінчення терміну та має дедуплікацію “1 раз на добу на препарат”.

## 2.4 Web-частина

- Працює вбудований web-клієнт у `wwwroot` (login/dashboard/medicine/iot/storage/audit тощо).
- Основні CRUD-сценарії реалізовані.

## 2.5 IoT + навантажувальні тести

- IoT емулятор (ESP32 + DHT22) реально читає сенсор (`dht.readTemperature()`, `dht.readHumidity()`), надсилає `POST /api/storagecondition` і підтягує пороги через `GET /api/iotdevice/{id}`.
- Є два NBomber-проєкти (`LoadTest.GET`, `LoadTest.POST`) для базових сценаріїв.

## 3) Як це реалізовано (архітектурно)

- **Backend:** `WebApp/MedicationManagement` (ASP.NET Core 8).
- **Дані:** SQL Server + EF Core, 2 контексти (`MedicineStorageContext`, `UserContext`).
- **Auth:** ASP.NET Identity + JWT.
- **Інтеграція процесів:** через DI, контролери → сервіси → EF Core.
- **Фонові процеси:** `BackgroundService` із `CreateScope()` для коректного scoped DbContext.
- **Зберігання enum:** як `string` у БД (через `HasConversion<string>()`).
- **Frontend:** поточний монолітний web у `wwwroot` (ще не SPA).
- **IoT:** PlatformIO/Wokwi (`IoTEmulate`), чутливі дані винесені в `src/config.h` (файл виключений із VCS).

## 4) Як система працює зараз (наскрізні процеси)

## 4.1 Авторизація і ролі

1. Користувач реєструється (`POST /api/auth/register`), перший користувач стає `Administrator`.
2. Вхід через `POST /api/auth/login` → JWT.
3. Захищені ендпоінти приймають Bearer token.
4. `GET /api/auth/me` повертає профіль і ролі поточного користувача.

## 4.2 Керування препаратами

1. Адміністратор створює/редагує/видаляє `Medicine`.
2. Доступні вибірки:
   - `low-stock`;
   - `expiring` (лише ті, що ще не прострочені, але вже в warning-вікні);
   - `replenishment-recommendations` (пороги з `Business` конфігурації).
3. Ключові дії логуються в `AuditLog`.

## 4.3 Моніторинг умов зберігання і інциденти

1. IoT надсилає вимірювання в `StorageCondition`.
2. `StorageConditionMonitoringService` періодично перевіряє останні значення активних пристроїв.
3. Якщо є порушення і немає активного інциденту типу `device+incidentType`:
   - створюється `StorageIncident (Active)`;
   - створюється `Notification`;
   - створюється `AuditLog (Warning)`.
4. Якщо параметри повернулися в норму:
   - інцидент закривається (`Resolved`);
   - створюється інформаційне сповіщення;
   - створюється `AuditLog (Info)`.

## 4.4 Lifecycle препарату

1. Через `POST /api/medicinelifecycle` додається подія (`Received`, `Issued`, `Moved`, `Expired`, `Disposed`, `Recalled`).
2. `PerformedBy` заповнюється з JWT-контексту.
3. Історія доступна загально або по конкретному препарату.

## 4.5 Сповіщення і аудит

- `Notification` використовується як структурований канал системних подій.
- `AuditLog` фіксує дії користувачів, сенсорів та фонових сервісів з деталями по сутності.

---

## 5) Поточна бізнес-логіка (суть)

- Препарат має життєвий цикл і може бути прив’язаний до локації зберігання.
- IoT-пристрій задає допустимі межі температури/вологості.
- Факт порушення не просто пишеться в лог, а оформлюється як окремий інцидент (`StorageIncident`) зі статусом.
- Система переходить від “сирих вимірювань” до “керованих подій і рішень”:
  - вимірювання → перевірка порогів → інцидент → сповіщення → аудит → закриття інциденту.
- Для запасів і термінів реалізовано автоматизовані управлінські підказки (`low-stock`, `replenishment`, `expiry warning`).

## 6) Чим поточний стан дипломного проєкту відрізняється від курсової

## Додано/покращено відносно курсової

- Повна дипломна доменна надбудова:
  - `StorageLocation`, `StorageIncident`, `MedicineLifecycleEvent`, `Notification`.
- Розширений `Medicine` і `AuditLog`.
- Нові API-контролери під ці сутності.
- Фонові служби переведені з простого логування на процесну модель інцидентів і сповіщень.
- Виправлені ключові ризики безпеки Фази 1 (JWT-пайплайн, авторизація `StorageCondition`, робота DHT22).
- Code review-доробки Фази 3.5: `AsNoTracking`, конфігурована бізнес-логіка, винесення чутливих IoT-параметрів, чистка архітектурних залишків у `Program.cs`.

## Що лишилось “спадком курсової” і ще не завершено

- Web frontend все ще вбудований у backend (`wwwroot`), окремого SPA ще немає.
- Токен у web зберігається в `localStorage`.
- CORS-політика для окремого SPA ще не увімкнена.
- Load tests все ще мають захардкоджені URL/токен.
- Відсутні unit/integration тести backend.
- Немає `docker-compose.yml`, `README.md`, `IMPLEMENTATION_ROADMAP.md`, архітектурних діаграм.
